## 1. FUNDAMENTOS DO PLANEJAMENTO MILITAR

### 1.1 Princípios Cardeais (Army Doctrine)

1. **Objetivo (Objective)** - Todo plano deve ter um objetivo final claro e mensurável
2. **Ofensiva (Offensive)** - Manter iniciativa sempre que possível
3. **Massa (Mass)** - Concentrar recursos no ponto decisivo
4. **Economia de Força (Economy of Force)** - Não desperdiçar recursos
5. **Maneuver (Maneuver)** - Ser flexível, não rígido
6. **Unidade de Comando (Unity of Command)** - Uma pessoa responsável
7. **Segurança (Security)** - Não dar informação ao "inimigo"
8. **Surpresa (Surprise)** - Agir quando menos esperado
9. **Simplicidade (Simplicity)** - Planos complexos falham em execução

### 1.2 O Ciclo OODA (John Boyd)

```
OBSERVE → ORIENT → DECIDE → ACT → (loop)
```

- **Observe**: Coleta de inteligência real-time
- **Orient**: Processar, filtrar, dar contexto
- **Decide**: Escolher o curso de ação
- **Act**: Executar

**Vantagem competitiva**: Completar o ciclo mais rápido que o oponente.

### 1.3 Classificação Nível de Operação

> **NOTA**: Esta nomenclatura foi atualizada para evitar confusão com o nome da empresa (L2 SYSTEMS).

| Nível | Código Antigo | Descrição | Perfil de Risco | Recursos Necessários |
|-------|--------------|-----------|-----------------|---------------------|
| **N1 (Básico)** | (era L1) | Operações rotineiras | Baixo | Mínimos |
| **N2 (Padrão)** | (era L2) | Operações sensíveis | Moderado-Alto | Especializados |
| **N3 (Avançado)** | (era L3) | Operações críticas | Crítico | Máxima mobilização |

**Este documento cobre N1 até N4 (todos os níveis).**

### 1.3.1 Mapeamento: Tipos de ROE vs Níveis de Operação

| ROE Tipo | Nível de Operação | Risco | Autorização | Exemplo de Uso |
|----------|-------------------|-------|-------------|----------------|
| **A** | N1 | Baixo | Automática | Abordagem inicial, teste de receptividade |
| **B** | N1-N2 | Baixo-Moderado | Operador | Proposta com chance de rejeição |
| **C** | N2-N3 | Alto | Chain of Command | Operação que requer resultado |
| **D** | N3-N4 | Crítico | Múltiplas aprovações | Stakes elevados, sem retorno |

#### Decisão: Qual ROE usar?

```
DETERMINAÇÃO DO TIPO DE OPERAÇÃO:

1. Qual o nível de risco?
   ├── Baixo → Tipo A
   ├── Moderado → Tipo B
   ├── Alto → Tipo C
   └── Crítico → Tipo D

2. Qual o nível de operação?
   ├── N1 (Básico) → Tipo A ou B
   ├── N2 (Padrão) → Tipo B ou C
   ├── N3 (Avançado) → Tipo C ou D
   └── N4 (Extremo) → Tipo D

3. Tenho autorização necessária?
   ├── Tipo A: Sim
   ├── Tipo B: Provavelmente
   ├── Tipo C: Verificar Chain of Command
   └── Tipo D: Requer aprovações múltiplas
```

## 1A. REGRAS DE ENGAJAMENTO (ROE)

### 1A.1 Princípios Fundamentais das ROE

```
LEGALIDADE → NECESSIDADE → PROPORCIONALIDADE → DISTINÇÃO
     │             │               │                  │
     ▼             ▼               ▼                  ▼
O ato é      A força é      O dano colateral    Separar alvos
legal?       necessária?     é justificável?     legítimos?
```

### 1A.2 Matriz ROE por Tipo de Operação

#### TIPO A: ABORDAGEM DIRETA (Aproximação Inicial)

| Parâmetro | Regra |
|-----------|-------|
| **Quando usar força?** | NUNCA. Força = falha imediata |
| **Quando recuar?** | Qualquer sinal de desconforto do alvo |
| **Trigger abortar** | Alvo demonstra rejeição ativa |
| **Nível de risco** | Baixo |
| **Prova requerida** | Nenhuma |
| **Autorização** | Automática (operador) |

```
ROE-TIPO-A:
├── AB: Imediato se alvo dizer "não", "pare", ou demonstrar desconforto
├── PROCEED: Até alvo demonstrar interesse
├── ESCALATE: NÃO ESCALAR - Mudar para Tipo B se necessário
└── DOCUMENT: Registrar apenas se legalmente necessário
```

> **EXEMPLO TIPO A:**
> - Situação: Você vê alguém interessante em um evento
> - Ação: Aproximar-se, iniciar conversa neutra
> - Se a pessoa demonstrar desconforto → Abort imediato
> - Se receptiva → Proceder para Tipo B se necessário

#### TIPO B: PROPOSTA ESTRATÉGICA (Proposta com risco de rejeição)

| Parâmetro | Regra |
|-----------|-------|
| **Quando usar força?** | NUNCA |
| **Quando recuar?** | Rejeição clara ou após 2 tentativas falhas |
| **Trigger abortar** | Bloqueio, rejeição explícita, silêncio >72h |
| **Nível de risco** | Moderado |
| **Prova requerida** | Justificativa clara do objetivo |
| **Autorização** | Operador com validação prévia |

```
ROE-TIPO-B:
├── ABORT: Rejeição explícita ou silêncio >72h
├── PROCEED: Resposta positiva ou neutro-ativo
├── DE-ESCALATE: Voltar para Tipo A
└── ESCALATE: NÃO - limite máximo deste tipo
```

> **EXEMPLO TIPO B:**
> - Situação: Você quer fazer uma proposta específica (pedir联系方式, sugerir encontro)
> - Ação: Fazer a proposta de forma clara
> - Se rejeição explícita ou silêncio >72h → Abort
> - Se resposta positiva → Proceder
> - Se neutro → De-escalate para Tipo A, construir rapport primeiro

#### TIPO C: OPERAÇÃO DE ALTO IMPACTO (Ação que requer resultados)

| Parâmetro | Regra |
|-----------|-------|
| **Quando usar força?** | Apenas se objetivo justifica E recursos permitem |
| **Quando recuar?** | Risco >Benefício OU chance sucesso <40% |
| **Trigger abortar** | Qualquer complicação imprevista |
| **Nível de risco** | Alto |
| **Prova requerida** | MOE claramente definido e mensurável |
| **Autorização** | Requiere validação de Chain of Command |

```
ROE-TIPO-C:
├── ABORT: 
│    ├── Chance sucesso <40%
│    ├── Risco pessoal >moderado
│    ├── Novo elemento de risco identificado
│    └── Recursos comprometidos
├── PROCEED:
│    ├── Chance sucesso >60%
│    ├── Risco <Benefício
│    └── Contingências prontas
├── DE-ESCALATE: Se contexto mudou negativamente
└── ESCALATE: Apenas para Tipo D com autorização
```

> **EXEMPLO TIPO C:**
> - Situação: Operação que requer resultado específico (fechar deal, obter compromisso)
> - Ação: Executar com todos os protocolos N2 ativos
> - Avaliar chance de sucesso: se <40% → Abort
> - Se chance >60% e contingênciasprontas → Executar
> - Requer Chain of Command approval

#### TIPO D: OPERAÇÃO CRÍTICA (Stakes elevados, sem retorno)

| Parâmetro | Regra |
|-----------|-------|
| **Quando usar força?** | Apenas se Existência/Nível muito alto em jogo |
| **Quando recuar?** | Qualquer dúvida sobre legalidade ou resultado |
| **Trigger abortar** | Qualquer anomalia no ambiente |
| **Nível de risco** | Crítico |
| **Prova requerida** | Análise completa de custo-benefício |
| **Autorização** | Múltiplas aprovações necessárias |

```
ROE-TIPO-D:
├── ABORT:
│    ├── Illegalidade mínima identificada
│    ├── Risco desproporcional
│    ├── Perda de suporte/logística
│    └── Elemento comprometido
├── PROCEED: Apenas se TODOS os critérios preenchidos
│    ├── Legalidade 100%
│    ├── Chance sucesso >70%
│    ├── Contingências testadas
│    └── Recursos assegurados
├── DE-ESCALATE: Preferível a abort quando possível
└── ESCALATE: Emergency only - não há Tipo E
```

> **EXEMPLO TIPO D:**
> - Situação: Stake muito alto, sem possibilidade de retorno (ex: decisão irreversível)
> - Ação: Múltiplas aprovações, análise completa de custo-benefício
> - NUNCA proceder se qualquer dúvida sobre legalidade
> - Chance de sucesso deve ser >70%
> - Requer validação de Chain of Command completo
> - De-escalate é preferível a abort quando possível

## 1B. FORCE PROTECTION (Proteção da Força)

### 1B.1 Níveis de Proteção

| Nível | Condição | Ação |
|-------|----------|------|
| **GREEN** | Ambiente seguro | Operação normal |
| **YELLOW** | Potencial ameaça identificada | Aumentar vigilância, preparar extração |
| **ORANGE** | Ameaça confirmada | Abortar operação, extração imediata |
| **RED** | Ameaça ativa | Evacuação de emergência, medidas de proteção |
| **BLACK** | Comprometimento total | Plano de negação, separação total |

### 1B.2 Protocolo de Extração

```
EXTRACTION PROTOCOL:
│
├── CONDIÇÃO: Comprometimento identificado
│
├── FASE 1: DETECTION (0-30s)
│   ├── Identificar tipo de comprometimento
│   ├── Avaliar severidade (1-5)
│   └── Alertar elementos envolvidos
│
├── FASE 2: MOBILITY (30s-5min)
│   ├── Rota primária de saída
│   ├── Rota secundária (backup)
│   ├── Rota terciária (emergência)
│   └── Ponto de regroup
│
├── FASE 3: DISPERSION (5-15min)
│   ├── Separar elementos
│   ├── Comunicação silenciosa
│   └── Destinos definidos por elemento
│
└── FASE 4: ASSESSMENT (15min+)
    ├── Verificar integridade de todos
    ├── Avaliar danos
    └── Decidir próximos passos
```

### 1B.3 Compromised Element Protocol

```
SE [ELEMENTO COMPROMETIDO]
ENTÃO:
   1. ISOLAR imediatamente do resto da operação
   2. NÃO CONTATAR outros elementos por 24-72h
   3. USAR canal de comunicação backup
   4. REAVALIAR Nível de ameaça da operação inteira
   5. SE ameaça ativa → ATIVAR plano de negação
   
ELEMENTO COMPROMETIDO NÃO PODE:
   ├── Aproximar-se de outros elementos
   ├── Acessar locais de operação
   ├── Usar canais primários de comunicação
   └── Executar qualquer operação ativa
```

### 1B.4 Battle Drills - Force Protection

```
DRILL 1: CONTACT LEFT/RIGHT
├── REAÇÃO: Mover-se perpendicular à ameaça
├── AÇÃO: Reportar, continuar ou abortar baseado em situação
└── PRIORITY: Sobreviver primeiro

DRILL 2: MEDICAL EMERGENCY
├── REAÇÃO: Primeiro elemento presta first aid
├── AÇÃO: others garantem perimeter
└── PRIORITY: Estabilizar, extrair

DRILL 3: COMPROMISE DISCOVERY
├── REAÇÃO: Freeze, avaliar, não demonstrar
├── AÇÃO: Extrair lentamente, não atrair atenção
└── PRIORITY: Negação se possível, abort se necessário
```

## 1D. MEDEVAC / CASUALTY MANAGEMENT

### 1D.1 Níveis de Casualty

| Nível | Descrição | Tempo | Ação |
|-------|-----------|-------|------|
| **MINOR** | Ferimento leve, ambulatório | 24h+ | Autotratamento |
| **SERIOUS** | Ferimento requer atenção médica | 6-24h | Medical attention required |
| **CRITICAL** | Vida em risco | <1h | Emergency extraction |
| **FATAL** | Perda humana | N/A | Protocolo de recuperação |

### 1D.2 Medical Extraction Plan

```
MEDEVAC PROTOCOL:
│
├── TRIGGER: Ferimento identificado
│
├── FASE 1: ASSESS (0-2min)
│   ├── Avaliar gravidade (MINOR/SERIOUS/CRITICAL)
│   ├── Aplicar primeiros socorros
│   └── Determinar se móvel
│
├── FASE 2: STABILIZE (2-15min)
│   ├── Controlar sangramento
│   ├── Imobilizar se necessário
│   └── Preparar para movimento
│
├── FASE 3: EXTRACT (15-60min)
│   ├── Rota para ponto médico
│   ├── Suporte durante transporte
│   └── Notificar receptor
│
└── FASE 4: TREATMENT (60min+)
    ├── Hand-off para profissional
    └── Report médico completo
```

### 1D.3 Pontos de Extração Médica

| Ponto | Local | Tempo Max | Capacidade |
|-------|-------|-----------|------------|
| **Ponto 1** | Mais próximo | 15min | Primeiros socorros |
| **Ponto 2** | Secundário | 30min | Atendimento básico |
| **Ponto 3** | Hospital/Clínica | 60min+ | Completo |

```

### 4.1 O Conceito de "Branch" e "Sequel"

- **Branch**: Plano alternativo para quando algo inesperado acontece DURANTE a operação
- **Sequel**: Plano para após completar a operação (o que fazer DEPOIS do sucesso/fracasso)

### 4.2 Estrutura de Contingência

```
SE [CONDIÇÃO X] ACONTECER
ENTÃO [AÇÃO Y]

SE Ela não responder em 48h
ENTÃO Enviar mensagem de follow-up com tom casual
PARA Manter a porta aberta sem parecer desesperado
```

## 5. PLANEJAMENTO DE "FALHA PLANEJADA"
## Operações de和信息 Gathering através de Estratégia de Perda Controlada

---

### 5.1 Fundamentos da "Strategic Failure"

Às vezes você **quer** perder uma batalha para ganhar a guerra. O conceito vai além de simplesmente "ser recusado" — é uma operação estruturada de **coleta de inteligência** e **engenharia psicológica**.

**Princípio Core**: A falha planejada é uma **técnica de reconhecimento** que revela informações sobre:
- Limites do alvo
- Padrões de reação sob pressão
- Indicadores de interesse genuíno vs. cortesia
- Vulnerabilidades psicológicas

---

### 5.2 Taxonomia de Falha Planejada

#### Tipo Alpha: "Feigned Approach" (Aproximação Simulada)

```
OBJETIVO: Testar receptividade sem exposição total
TÉCNICA: Apresentar interesse moderado, recolher resposta
FALHA: Recusa suave (simulada)
INFO COLETADA: Indicadores de interesse

APLICAÇÃO:
- Iniciar conversa sobre tema neutro
- Introduzir elemento de interesse próprio subtle
- Observar reação (aproximação vs. afastamento)
- Recuar se interesse detectado → mudar para abordagem direta
- Recuar se desinteresse → mudança de contexto
```

#### Tipo Beta: "Commitment Probe" (Teste de Compromisso)

```
OBJETIVO: Criar micro-compromissos, medir compliance
TÉCNICA: Pedir pequena coisa primeiro, escalar gradualmente
FALHA: Recusa no pedido pequeno
INFO COLETADA: Limite de compliance, padrões de resposta

APLICAÇÃO (Foot-in-the-Door modificado):
- Pedir informação não-sensível primeiro
- Se concedido → pedir algo maior
- Se recusado → "OK, sem problemas" (aceitar gracefulmente)
- Observar padrão: cedência → resistência → fronteira
- Mapear a fronteira entre receptividade e rejeição
```

#### Tipo Gamma: "Cognitive Dissonance Inducer"

```
OBJETIVO: Criar desconforto psicológico, observar reação
TÉCNICA: Apresentar informação que desafia percepção do alvo
FALHA: Negação ou rejeição da informação
INFO COLETADA: Como o alvo lida com dissonância, valores revelados

APLICAÇÃO:
- Introduzir perspectivacontrária à posição do alvo
- Não defender a perspectiva, apenas apresentar
- Observar: concordânciareluctante? defensividade? mudança de tema?
- Não pressionar por resposta → deixar fermentar
```

#### Tipo Delta: "Social Proof Test"

```
OBJETIVO: Testar reação a terceiros/influência social
TÉCNICA: Mencionar outros interessando mesmo alvo
FALHA: Desinteresse demonstrado
INFO COLETADA: Nível de interesse real, ciúme/indiferença

APLICAÇÃO:
- Mencionar (verdade ou fiction) que outras pessoas demonstraram interesse
- Observar reação: curiosidade? indiferença? competição?
- NÃO mencionar nomes específicos
- Usar linguagem vaga "as pessoas parecem notar..."
```

#### Tipo Epsilon: "Value Disqualifier"

```
OBJETIVO: Testar se alvo se qualifica PARA VOCÊ
TÉCNICA: Apresentar "defeito" próprio ou situação
FALHA: Alvo usa como desculpa para recusar
INFO COLETADA: Prioridades do alvo, superficial vs. essência

APLICAÇÃO:
- Mencionar algo que poderia ser "problema" (trabalho, agenda, etc.)
- Se o alvo rápidamentepega isso como reason para distância:
  → Revela que foca em Conveniência, não Conexão
- Se o alvo ignora e continuainteressado:
  → Revela interesse na pessoa, não na imagem
```

#### Tipo Zeta: "Urgency Reversal"

```
OBJETIVO: Inverter script esperado, observar quem faz trabalho
TÉCNICA: Apresentar que você está "indisponível" temporariamente
FALHA: Alvo não demonstra interesse em resgataroportunidade
INFO COLETADA: Quem está realmenteinteressado

APLICAÇÃO:
- Mencionar que você terá menos tempo/disponibilidade
- NÃO propor alternativa → esperar reação
- Se o alvo propõemeeting alternativo:
  → Interesse genuíno
- Se o alvo simplesmenteaceita a redução:
  → Interesse superficial
```

### 5.3 Técnicas Psicológicas Aplicadas

#### 5.3.1 Reciprocidade Invertida

```
PRINCÍPIO: Oferecer valor primeiro, coletar informação depois

APLICAÇÃO:
- Dar informação útil/serviço kecilho
- Quando o alvo se sentir "em dívida":
  → Fazer pergunta estratégica
  → Pedir feedback sobre si mesmo
  → Observar o que o alvo escolhe revelar

FALHA: Recusaro "débtot" Offered → não há reciprocidade
RESULTADO: Mapeamento de disposição para reciprocidade
```

#### 5.3.2 Low-Ball Strategy (Isca Baixa)

```
PRINCÍPIO: Oferecer algo atrativo, depois revelar "catch"

APLICAÇÃO:
- Propor atividade muito atraente
- Quando alvo aceita → adicionar condição/minor "problema"
- Observar:
  - Se mantém aceitação → interesse forte
  - Se recua → interesse condicional

FALHA: Alvo recua quando condição adicionada
INFO: Interesse era no benefício, não na pessoa/atividade
```

#### 5.3.3 Scarcity & Urgency Manipulation

```
PRINCÍPIO: Criar janela de oportunidade, medir reação

APLICAÇÃO:
- Mencionar que "oportunidade" está fechando (sem mentir)
- NÃO PRESSIONAR → deixar espaço
- Observar se alvo:
  - Pega a deixa e agenda → interesse ativo
  - Deixa passar → interesse passivo/ausente
  - Questiona a urgência → cético (inteligente)

FALHA: Janela fecha sem ação do alvo
INFO: Não era prioritário suficiente
```

#### 5.3.4 Authority & Similarity Testing

```
PRINCÍPIO: Demonstrar expertise/autoridade em área, observarattraction

APLICAÇÃO:
- Demonstrar conhecimento genuíno em algo relevante
- NÃO ALARDAR → mencionar naturalmente
- Observar se alvo:
  - Pergunta mais → compatibility intelectual
  - Desvia → não tem interesse em profundidade
  - Tenta impressionar de volta → mutual interest

FALHA: Alvo não se engaja no tópico
INFO: Falta de conexão intelectual
```

### 5.6 Modelo de "Controlled Loss" Avançado

```
INVESTIMENTO → RISCO → RETORNO ESPERADO → DECISÃO

MATRIZ DE DECISÃO:

Se RISCO < 20% E INFO ALTA → EXECUTAR
Se RISCO 20-40% E INFO MÉDIA → CONSIDERAR
Se RISCO > 40% → NÃO EXECUTAR (mudar tipo)

CÁLCULO DE EXPECTED VALUE (EV) MELHORADO:

EV = (Prob_Sucesso × Valor_Info) - (Prob_Falha × Custo_Falha)

Exemplo:
- Prob_Sucesso (info revelada): 60%
- Valor_Info: 8 (alta)
- Prob_Falha (relação danificada): 40%
- Custo_Falha: 3 (baixo)

EV = (0.6 × 8) - (0.4 × 3) = 4.8 - 1.2 = 3.6 (POSITIVO → EXECUTAR)
```

### 5.7 Técnicas de Recuperação Pós-Falha

#### 7.1 "Graceful Withdrawal"

```
Se falha occurred:
├── NÃO EXPLICAR ou justificar
├── NÃO ficar em silêncio absoluto (parece afetado)
├── Aceitar com naturalidade:
│   ├── "Ah, sem problemas!"
│   ├── "Entendi, talvez outra hora"
│   └── Mudar tópico leve
└── Manter interação normal por curto período depois

REGRA: A recuperação diz mais sobre você do que a falha
```

#### 7.2 "Escalation Through Acceptance"

```
Se falhaaccepted gracefully:
├── O alvo pode se sentir guilty
├── Usar isso para:
│   ├── Pedir algo em troca (pequeno)
│   └── Criar "débitot" psikologis
│   └── Reargument não necessário → deixe-o pensar
└── Não exploitar → maintain respect
```

#### 7.3 "Strategic Patience"

```
Pós-falha:
├── NÃO fazer follow-up imediato
├── Esperar que ALVO faça trabalho
│   ├── Se interessadocontatará
│   └── Se não, não era recíproco
└── Tempo mínimo: 72h (regra ouro)

Se alvo não contatar em 72h:
├── Avaliação: interesse era unidirecional
└── Mover para outro alvo ou re-approach months later
```

#### Caso 3: "Relação em Progresso"

FALHA: Dismissal completo → não há depth de conexão
RESULTADO: Engagement → há potential
```

## 12. FÓRMULAS ÚTEIS

### Probabilidade Condicional
```
P(A|B) = P(A e B) / P(B)
Exemplo: P(Proposta aceita | Alvo está disponível) = ?
```

### Expected Value
```
EV = (Probabilidade de Sucesso × Valor do Sucesso) - (Probabilidade de Falha × Custo da Falha)
Exemplo: EV = 0.4 × 100 - 0.6 × 20 = 28 (positivo, vale a pena)
```

### Risco/Ação
```
Se EV > 0 → Executar
Se EV < 0 → Não executar ou ajustar
Se EV ≈ 0 → Executar apenas se estratégica (não por perda)
```
